# P24.1 Runtime Hardening Design Bridge

## Scope

P24.1 records the current runtime-hardening evidence as a bounded report:

- schema `tetra.runtime.hardening.v1`
- scope `p24.1_runtime_hardening`
- one row for each master-plan target: deterministic traps, OOM policy, stack
  overflow guard, integer overflow semantics audit, allocator corruption
  instrumentation, region double-free/use-after-free instrumentation, actor
  mailbox overflow policy, and network parser limits

The report is evidence-backed inventory plus validator gates. It must not
change runtime behavior, safe-program semantics, allocation legality, bounds
semantics, overflow semantics, or unsafe policy.

## Live Witnesses

- `runtimeabi.RuntimeAllocationContracts` validates allocation failure behavior,
  negative-size guards, byte-size overflow guards, report hooks, and debug
  instrumentation such as `bounds_header`, `double_free`, and `use_after_free`.
- `runtimeabi.RuntimeRegionAllocatorConfig`, `runtimeabi.AlignRegionBytes`,
  `runtimeabi.RuntimePerCoreSmallHeapABI`, and
  `runtimeabi.NewPerCoreSmallHeapAllocator` provide region/small-heap alignment,
  max payload, reuse metadata, stale-handle, and double-free evidence.
- `parallelrt.NewTypedMailbox` provides bounded mailbox capacity,
  `blocking_recv_yield` backpressure metadata, FIFO receive, and recoverable
  `ErrMailboxFull` evidence.
- `actorsrt.ActorRuntimeProductionBoundaryAudit` records the production actor
  runtime boundary, including the current blocker that built-in message-pool
  overflow is not yet a checked runtime error.
- `httprt.ParseRequest` and `httprt.ParseRequestView` enforce header, header
  count, body, malformed request/header, unsupported version, and unsupported
  transfer-encoding limits.
- `pgrt.ReadFrame` enforces PostgreSQL frame length and max payload checks with
  deterministic malformed/too-large errors.
- Optimizer coverage and constant-expression diagnostics preserve integer
  overflow semantics by rejecting unsafe `checkedNegI32` and
  `foldConstBinaryI32` folds rather than rewriting them.
- Backend stack-depth checks and wasm trap emitters are reviewed as deterministic
  trap/stack-shape evidence, while full guard-page or recursion-depth runtime
  stack-overflow protection remains a non-claim.

## Rows

1. `deterministic_traps`
2. `oom_policy`
3. `stack_overflow_guard`
4. `integer_overflow_semantics_audit`
5. `allocator_corruption_detection`
6. `region_double_free_use_after_free`
7. `actor_mailbox_overflow_policy`
8. `network_parser_limits`

## Validator Duties

The P24.1 validator must reject:

- missing rows, duplicate rows, missing witnesses, or placeholder evidence;
- missing review flags for any master-plan target;
- fake full runtime-hardening claims;
- fake full OOM recovery, full stack-overflow protection, full allocator
  corruption detection, or production actor-mailbox claims;
- runtime behavior, safe-semantics, or performance claims;
- missing non-claims for the bounded runtime-hardening scope.

## Non-Claims

- No full runtime-hardening proof is claimed.
- No full stack-overflow protection is claimed.
- No OOM recovery guarantee is claimed.
- No full allocator-corruption detection proof is claimed.
- No production actor-mailbox promotion is claimed.
- No runtime behavior change is claimed.
- No safe-program semantics change is claimed.
- No performance claim is made.
