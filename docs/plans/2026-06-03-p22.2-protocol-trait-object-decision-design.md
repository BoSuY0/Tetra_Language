# P22.2 Protocol / Trait Object Decision Design

Date: 2026-06-03
Slice: P22.2 Protocol / Trait Object Decision

## Intent

P22.2 answers the master-plan question: Tetra keeps the current static
protocol/conformance fast path for this branch and does not promote runtime
protocol values, trait objects, witness tables, or dynamic dispatch without a
separate same-branch ABI/lifetime design.

The existing implementation already supports:

- static `protocol` declarations and `impl Type: Protocol` conformance checks
  in `compiler/internal/semantics/checker.go`;
- extension/static method signature, ownership, effects, `throws`, `async`,
  and MVP generic-requirement shape comparison through
  `compareProtocolRequirement`;
- static protocol-bound generic validation during monomorphization through
  `validateGenericFuncDecl`;
- explicit rejection for generic-bound requirement calls such as `T.echo(x)`;
- explicit absence of protocol runtime values because protocols are tracked
  separately from value types;
- P17.2/P21.2 specialization evidence for known direct static
  protocol/conformance calls.

The missing P22.2 piece is a decision report and validator that make this
choice durable and reject fake promotion claims.

## Report Shape

Add a compiler-level report:

- schema: `tetra.language.protocol_trait_object_decision.v1`
- scope: `p22.2_protocol_trait_object_decision`
- decision: `keep_static_conformance_only`
- rows:
  - `static_conformance_fast_path`
  - `static_protocol_bound_generics`
  - `runtime_existential_decision`
  - `explicit_dynamic_dispatch_gate`
  - `specialization_static_abstraction`
  - `witness_table_boundary`
  - `trait_object_boundary`
  - `registry_docs_alignment`

The report should include concrete witnesses built from real compiler APIs:

- parse/check/lower a protocol impl with a direct extension method call and
  verify the lowered IR uses a known direct call target;
- parse/check/lower a protocol-bound generic call and verify the concrete
  monomorphized signature and direct call target;
- verify runtime protocol values remain rejected as an unknown type;
- verify generic-bound requirement calls remain rejected with the stable
  unsupported-MVP diagnostic;
- read P17.2/P21.2 specialization report rows to prove the existing static
  abstraction-removal evidence is bounded to known direct symbols and no
  dynamic dispatch/witness-table removal is claimed.

## Validator Guardrails

The validator must reject:

- missing or duplicate rows;
- placeholder evidence;
- missing witness coverage;
- runtime existential promotion claims;
- trait-object promotion claims;
- witness-table promotion claims;
- dynamic-dispatch promotion claims;
- conformance-table lookup promotion claims;
- runtime protocol value claims;
- broad specialization claims;
- performance claims;
- runtime behavior changes;
- safe-program semantic changes.
