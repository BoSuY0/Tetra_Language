# P23.0 Translation Validation v2 Design

Status: bridge before RED/GREEN.

## Goal

Close P23.0 as evidence-backed translation-validation v2 for the currently
supported optimizer subset. The slice strengthens and reports the existing
`validation.ValidateTranslation` / `opt.Manager` contract; it does not add a
public optimizer mode or claim a full formal proof.

Schema: `tetra.translation.validation.v2`

Scope: `p23.0_translation_validation_v2`

## Evidence Sources

- `compiler/internal/validation/validation.go` for verifier checks,
  proof-fact multiset comparison, symbolic scalar equivalence, differential
  samples, and `BuildOptimizationValidationMetadata` sha256 before/after
  hashes.
- `compiler/internal/opt/manager.go` for registered pass metadata,
  `validation.ValidateTranslation` hook enforcement, pass reports, and
  validation metadata.
- `compiler/internal/differential/differential.go` for source/Stack IR/
  optimized Stack IR/SSA/Machine/native lanes across the supported scalar,
  loop, call, and i32 slice memory subset.
- `compiler/internal/allocplan/plan.go` plus
  `validation.ValidateAllocationLowering` for allocation-plan preservation
  evidence.

## Coverage Rows

- `registered_passes`: every registered optimizer pass uses the translation
  validation strategy and carries report/metadata rows.
- `symbolic_scalar_equivalence`: scalar arithmetic and comparison rewrites are
  symbolically checked and sampled.
- `memory_equivalence`: supported stack/local i32 slice load/store samples are
  compared through the stable interpreter subset.
- `bounds_proof_preservation`: proof-tagged removed bounds checks preserve the
  proof multiset and reject missing or changed proof ids.
- `allocation_plan_preservation`: allocation lowering evidence remains checked
  against allocation-plan rows.
- `machine_checkable_hashes`: optimization metadata records sha256 before/after
  IR hashes and compared functions.

## Non-claims

- No full formal proof of the language.
- No exhaustive optimizer-pass completeness beyond the registered supported
  subset.
- No broad memory model, alias model, or loop theorem prover.
- No performance claim.
- No runtime behavior change.
- No safe-program semantics change.
