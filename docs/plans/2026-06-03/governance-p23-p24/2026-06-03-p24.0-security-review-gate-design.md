# P24.0 Security Review Gate Design

Status: active implementation bridge for the Ideal Master Plan P24.0 slice.

## Purpose

P24.0 adds an evidence-backed security review gate for the current repository
state. It is a compiler/reporting and documentation gate, not a release
security signoff and not a production security certification.

The gate records schema `tetra.security.review_gate.v1` and scope
`p24.0_security_review_gate`.

## Required Surfaces

The report must cover every P24.0 surface from the master plan:

- unsafe APIs
- capabilities
- memory allocator
- network runtime
- actor runtime
- DB protocol
- package/Eco system
- build scripts
- supply chain

Each surface must have at least one row with concrete evidence paths, a focused
test or validator command, boundary text, and witness linkage.

## Required Artifacts

The P24.0 artifacts live under `docs/audits/`:

- `docs/audits/security-review.md`
- `docs/audits/threat-model.md`
- `docs/audits/unsafe-surface-map.md`
- `docs/audits/capability-surface-map.md`

These are durable review artifacts for the current branch state. They do not
replace `scripts/release/v1_0/security-review.sh`, which remains the named
reviewer release-signoff validator and requires a release report directory with
artifact hashes and residual-risk decision fields.

## Live Witnesses

The report builder reuses current validated surfaces where possible:

- `runtimeabi.RuntimeAllocationContracts` and
  `runtimeabi.RuntimeRawPointerBoundsABI`
- `netrt.IOReactorCoverage` and `netrt.ValidateIOReactorCoverage`
- `actorsrt.ActorRuntimeProductionBoundaryAudit` and its validator
- `pgrt.ProductionPostgresCoverage` and its validator
- concrete spec/tool paths for unsafe, capability, Eco, release scripts, and
  supply-chain rows

## Non-Claims

P24.0 must reject fake promotion claims:

- no security certification
- no external penetration test
- no CVE-free claim
- no release security signoff claim
- no runtime behavior change
- no safe-program semantics change
- no performance claim

Future release signoff can cite these artifacts as input, but it must still
run the release signoff validator in the release artifact archive.
