# P21.1 ABI Verification v1 Design

Status: design checkpoint for the Ideal Master Plan goal loop on 2026-06-03.

## Inputs

- Master plan slice: P21.1 ABI Verification v1.
- Required targets: linux-x64 SysV, linux-x86, linux-x32, macos-x64,
  windows-x64, wasm32-wasi, and wasm32-web.
- Required task coverage: ABI test corpus, struct/enum/slice/String return
  validation, call boundary validation, and FFI repr(C) tests.
- Graphify MCP context:
  - `query_graph` identified existing ABI suite and ABI testcase nodes.
  - `get_neighbors("RunTargetABIChecks")` linked `compiler/abi_suite.go`,
    x86/x32/x64 suite runners, and the CLI `--abi` path.
  - `shortest_path("TestRunTargetABIChecksCoversX86X64AndRejectsWASM",
    "wasm32-wasi")` found no target node, matching the missing wasm suite.

## Existing Evidence To Reuse

- Native targets already have target-model checks, x86 i386 SysV
  classification, x64 SysV/Win64/x32 classification, varargs/sret and
  aggregate checks, object smoke checks, and repr(C)-gated exported aggregate
  diagnostics.
- WASM targets already define ILP32 target metadata and backend object/link
  validators for function metadata, local slots, symbol tokens, and `IRCall`
  arg/return slot matching.
- P21.0 already established that default structs are compiler-owned layout and
  exported FFI aggregate boundaries require explicit `repr(C)`.

## Implementation Plan

1. Extend `RunTargetABIChecks` with a wasm32 route.
2. Add wasm ABI checks that verify:
   - target identity and ILP32 wasm slot model;
   - struct/enum/slice/String return layout sizes and alignments;
   - `IRCall` boundary validation through wasm object/link metadata checks;
   - the FFI boundary remains honest: native exported aggregates require
     explicit `repr(C)`, while wasm checks validate compiler-owned slot ABI and
     do not claim a native C aggregate ABI.
3. Add an in-repo ABI verification report schema:
   - `tetra.abi.verification.v1`;
   - scope `p21.1_abi_verification`;
   - one row for each required target;
   - explicit task coverage rows and non-claims.
4. Document the audit and link it from the feature registry and manifest.

## Non-Claims

- No runtime execution claim for build-only native targets or wasm targets.
- No C ABI claim for default structs.
- No native C aggregate ABI claim for wasm targets.
- No performance claim.
- No safe-program semantics change.
