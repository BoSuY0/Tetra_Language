# Performance Blocker Reports V1

Status: P20.1 bounded evidence slice.

This audit records the compiler performance-blocker explanation contract for the master-plan P20.1
gate. It proves actionable explanation coverage for the P20.0 Tetra benchmark matrix. It is not
measured performance evidence, not C/C++/Rust parity, not an official benchmark result, not an
official TechEmpower result, not Tier 1+ P20.2 claim-tier evidence, and not an optimizer or runtime
behavior change.

## Report Surface

The compiler `--explain` path emits:

- `app.perf.json`
- schema version `3`
- kind `perf`
- matrix scope `p20.0_benchmark_matrix`
- matrix report `reports/benchmark-matrix-hardening-v1/benchmarks/p20-matrix-hardening-report.json`

The implementation and validator live in:

- `compiler/compiler_reports.go`
- `buildPerformanceReport`
- `ValidatePerformanceBlockerReport`

## Canonical Blocker Reasons

The report requires the exact master-plan reasons:

- `left bounds check: missing dominance`
- `heap allocation: escapes through return`
- `heap allocation: unknown call`
- `not vectorized: no noalias proof`
- `not inlined: code-size budget`
- `register spill: live range pressure`
- `stack fallback: unsupported aggregate return`
- `actor copy: borrowed data crosses boundary`

Every blocker row records a stable code, compiler component, `cost_class`, evidence source, and next
step. The validator rejects missing blocker rows, unknown blocker codes, unknown or drifted cost
classes, message drift, and placeholder evidence or next steps.

The canonical memory-relevant cost classes are:

- `bounds.missing_dominance`: `dynamic_check_required`
- `allocation.return_escape`: `conservative_fallback`
- `allocation.unknown_call`: `conservative_fallback`
- `vector.no_noalias_proof`: `dynamic_check_required`
- `stack_fallback.unsupported_aggregate_return`: `conservative_fallback`
- `actor_copy.borrowed_data_boundary`: `conservative_fallback`

Non-memory explanation blockers use `instrumentation_only` unless a later validated memory fact
promotes them.

## Validation

`ValidatePerformanceBlockerReport` rejects:

- missing schema, target, matrix scope, or matrix report metadata
- missing or duplicate blocker rows
- drift from the exact master-plan blocker messages
- missing or duplicate P20.0 benchmark explanation rows
- benchmark rows that cite unknown reason codes
- weak placeholder explanation or next-step text
- fastest-language, C++/Rust parity, official benchmark, official TechEmpower, measured-speed,
  throughput/latency advantage, or runtime-behavior-change claims unless explicitly written as
  non-claims
- dynamic_check_required zero-cost claims and trusted unsafe_unknown optimization claims

## Boundaries

- The report explains blockers; it does not remove them.
- The report is tied to the P20.0 dry-run matrix and does not imply benchmark execution.
- P20.2 owns public claim tiers and wording for measured performance. Current P20.1 blocker
  explanations validate only Tier 0 local-smoke wording in
  `reports/claim-tiers-v1/claim-tier-report.json`.
- No flag or report changes safe-program semantics.
